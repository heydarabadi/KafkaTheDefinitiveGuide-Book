configuring Apache Kafka and Apache ZooKeeper (required in versions up

### Operating System

-   Recommended for production: Linux
-   Installation steps in this chapter target Linux

### Java Requirements

## 2. Installing & Configuring ZooKeeper

### Purpose

ZooKeeper stores Kafka cluster metadata (brokers, topics, partitions,
consumer group info, offsets --- until KRaft mode replaces it in later
versions).

### Standalone Mode (Development / Testing)

``` bash
tar -zxf apache-zookeeper-3.5.9-bin.tar.gz
mv apache-zookeeper-3.5.9-bin /usr/local/zookeeper
mkdir -p /var/lib/zookeeper

cat > /usr/local/zookeeper/conf/zoo.cfg << EOF
tickTime=2000
dataDir=/var/lib/zookeeper
clientPort=2181
EOF

export JAVA_HOME=/usr/java/jdk-11.0.10
/usr/local/zookeeper/bin/zkServer.sh start
```

Verify:

``` bash
telnet localhost 2181
# type: srvr
```

### Production Ensemble

-   Use odd number of nodes: 3, 5 or 7
-   Sweet spot: 5 nodes → tolerates 2 failures

Example zoo.cfg:

``` properties
tickTime=2000
dataDir=/var/lib/zookeeper
clientPort=2181
initLimit=20
syncLimit=5
server.1=zoo1.example.com:2888:3888
server.2=zoo2.example.com:2888:3888
server.3=zoo3.example.com:2888:3888
```

Each server must have a unique myid file in its dataDir.

Best practice:

    zookeeper.connect=zoo1:2181,zoo2:2181,zoo3:2181/kafka-prod

## 3. Installing & Starting a Kafka Broker

``` bash
tar -zxf kafka_2.13-2.7.0.tgz
mv kafka_2.13-2.7.0 /usr/local/kafka
mkdir /tmp/kafka-logs

export JAVA_HOME=/usr/java/jdk-11.0.10
nohup /usr/local/kafka/bin/kafka-server-start.sh /usr/local/kafka/config/server.properties &
```

Validation:

``` bash
kafka-topics.sh --bootstrap-server localhost:9092 --create --topic test --partitions 1 --replication-factor 1
kafka-topics.sh --bootstrap-server localhost:9092 --describe --topic test
kafka-console-producer.sh --bootstrap-server localhost:9092 --topic test
kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic test --from-beginning
```

## 4. Key Broker Configuration Parameters — Detailed Explanation

Kafka brokers have several configuration parameters that are critical for cluster stability, durability, and performance. Here's a detailed look at each:

| Parameter | Purpose & Explanation |
|-----------|---------------------|
| **broker.id** | Each broker must have a unique ID within the cluster. Used for leader election, replica assignment, and broker identification. Reusing IDs can cause cluster conflicts and instability. |
| **listeners** | Defines the endpoints where the broker accepts client and inter-broker connections. Example: `PLAINTEXT://0.0.0.0:9092`. Multiple listeners can be configured for internal/external traffic or SSL/SASL. Proper configuration is essential for security and network separation. |
| **zookeeper.connect** | Connection string to the ZooKeeper ensemble (e.g., `zoo1:2181,zoo2:2181,zoo3:2181/kafka-prod`). Kafka relies on ZooKeeper for cluster metadata, leader election, and consumer offsets (pre-KRaft). Using a chroot path allows multiple Kafka clusters to share the same ZooKeeper safely. |
| **log.dirs** | Specifies one or more directories where Kafka stores its log segments. Multiple directories improve I/O performance and redundancy. Ensure enough disk space for retention, replication, and future growth. |
| **auto.create.topics.enable** | If `true`, Kafka automatically creates topics when a producer or consumer references a non-existent topic. For production, set to `false` to avoid accidental topic creation and ensure controlled configuration of partitions and replication. |
| **num.partitions** | Default number of partitions for newly created topics (when `auto.create.topics.enable` is true). Determines parallelism for producers and consumers. In production, explicitly create topics with desired partitions instead of relying on default. |
| **default.replication.factor** | Default replication factor for newly created topics. Ensures durability by storing multiple copies of each partition across brokers. Production clusters usually use ≥3 replicas for fault tolerance. |
| **log.retention.ms** | Time-based retention for log segments. Kafka will delete messages older than this period. Useful for controlling storage usage and compliance with retention policies. |
| **log.retention.bytes** | Size-based retention per partition. Kafka will delete older segments if the total size exceeds this limit. Can be combined with time-based retention. `-1` disables size-based deletion. |
| **min.insync.replicas** | Minimum number of replicas that must acknowledge a write for it to be considered successful (used with `acks=all`). Ensures durability during broker failures and prevents data loss. Typically ≥2 in production. |
| **message.max.bytes** | Maximum size of a single message (compressed or uncompressed). Must match producer and consumer fetch configurations. Prevents oversized messages from causing replication or memory issues. |

### Key Takeaways
- These parameters control **durability, availability, and throughput** of your Kafka cluster.
- Misconfigurations can lead to data loss, broker instability, or high latency.
- For production clusters, carefully tune:
    - replication factor
    - min.insync.replicas
    - log retention policies
    - disk allocation (`log.dirs`)

## 5. Hardware Selection Priorities — Detailed Explanation

Kafka is primarily a disk- and network-bound system. Performance depends far more on I/O and OS page cache than on CPU power.

### Disk throughput is most critical
Kafka is a distributed commit log that writes and reads data from disk continuously.

Every message:
1. Is appended to a log file on disk
2. Is replicated to follower brokers
3. Is read by consumers from disk (often via page cache)

Why disk throughput matters:
- Directly impacts producer latency
- Controls replication speed and stability
- Affects consumer lag under load
- Determines recovery speed after broker restart
- Log compaction is highly disk-intensive

Key characteristics:
- Kafka uses sequential I/O
- Throughput is more important than IOPS
- Slow disks cause replication lag and ISR shrink

Production insight:
If disk is saturated, adding CPU or RAM will not improve throughput.



### SSD preferred
SSDs provide lower latency and more consistent throughput compared to HDDs.

Benefits:
- Faster sequential writes
- Faster replication catch-up
- Better performance during traffic bursts
- Improved broker recovery time
- More predictable performance under load

When HDD may be acceptable:
- Long-term retention clusters
- Low-throughput environments
- Cost-optimized storage tiers

Rule of thumb:
SSD → performance clusters  
HDD → archival or cold storage clusters


### High network bandwidth (≥ 10 Gbit)
Kafka generates heavy internal and external network traffic.

Network usage comes from:
- Producers sending messages to brokers
- Broker-to-broker replication
- Consumers fetching data
- Metadata synchronization

Why high bandwidth is required:
- Replication duplicates full message payloads
- Many partitions replicate in parallel
- Consumer groups can read at maximum speed
- Cross-rack replication multiplies traffic

Symptoms of insufficient network capacity:
- Replication lag increases
- ISR instability
- Uneven broker load
- Throughput throttling under peak load

Production baseline:
10 Gbit network recommended  
Large clusters often require 25–40 Gbit

### Large memory for page cache
Kafka relies heavily on the Linux OS page cache rather than JVM heap memory.

How memory is used:
- Data written to disk is cached by the OS
- Consumers often read directly from memory
- Replication fetches are frequently served from cache

Why more RAM improves performance:
- Higher cache hit rate
- Fewer physical disk reads
- Lower read latency
- Higher sustained throughput

Typical production memory strategy:
- JVM heap: 4–8 GB
- Remaining system memory: page cache
- Avoid oversized JVM heap

Bad configuration pattern:
Large heap + low OS memory → poor Kafka performance

### CPU is secondary
Kafka performs minimal computation compared to its I/O workload.

CPU is mainly used for:
- Network request handling
- Compression and decompression
- Log compaction
- Encryption (TLS)
- Partition management

CPU becomes important when:
- Heavy compression is enabled
- TLS is used everywhere
- Very high partition counts exist
- Brokers run additional processing workloads

In most deployments:
Disk or network saturates before CPU.

### Practical investment priority

1. Fast disks (SSD, high throughput)
2. High-bandwidth network
3. Large system memory (page cache)
4. CPU capacity

This priority order reflects real-world Kafka production bottlenecks.

## 6. Cluster Sizing Guidelines — Detailed Explanation

Proper cluster sizing ensures stability, fault tolerance, and predictable performance under load. Kafka scalability is primarily constrained by **partition count, replication load, and broker capacity**.



### Production minimum: 3–5 brokers

Kafka relies on replication for durability and availability.  
With fewer than 3 brokers, fault tolerance is severely limited.

Why 3 brokers is the minimum:
- Supports replication factor = 3
- Allows one broker failure without data loss
- Enables leader re-election without downtime
- Maintains quorum-based durability guarantees

Why 5 brokers is common in production:
- Tolerates multiple failures
- Enables rolling maintenance with lower risk
- Provides better partition distribution
- Improves load balancing

Operational insight:
More brokers increase availability, but also increase coordination overhead. Scale gradually based on throughput and storage needs.



### < 14k partition replicas per broker

This is a practical scalability guideline, not a hard technical limit.

What counts as a replica:
Each partition copy stored on a broker counts as one replica.

Example:
- 1 topic
- 100 partitions
- replication factor = 3
  → 300 partition replicas across the cluster

Why replica count matters:
Each replica consumes:
- file handles
- memory metadata
- network connections
- disk I/O bandwidth
- controller coordination overhead

Too many replicas per broker leads to:
- slow leader elections
- long startup and recovery time
- controller overload
- unstable cluster behavior
- increased GC pressure

Production guidance:
Keep partition count proportional to:
- disk throughput
- available memory
- expected consumer parallelism

Scaling strategy:
If partition count grows, add brokers instead of overloading existing ones.

---

### Enable rack awareness

Rack awareness helps Kafka place replicas across failure domains.

Failure domain examples:
- physical rack
- availability zone
- data center power group
- cloud availability zone

Why rack awareness is critical:
Without it, Kafka may place multiple replicas on the same rack.  
A single rack failure could then remove all replicas of a partition.

With rack awareness enabled:
- leader and followers are distributed across racks
- cluster survives rack-level outages
- replication traffic is balanced across network paths

Configuration concept:
Each broker is assigned a location label, and Kafka spreads replicas across those locations.

Benefits:
- higher availability
- predictable failover behavior
- safer replication strategy
- better cross-rack load distribution


### Key takeaway

Cluster sizing is not only about storage capacity — it is about balancing:
- replication overhead
- partition scalability
- fault tolerance
- operational stability

Safe production strategy:
Scale horizontally (more brokers) rather than vertically (overloading a broker).

## 7. Linux OS & JVM Tuning — Detailed Explanation

Kafka performance depends heavily on Linux I/O behavior and memory management.  
These settings help maximize disk throughput, stabilize write performance, and reduce GC impact.

### Virtual Memory

#### vm.swappiness = 1
Controls how aggressively Linux moves memory pages from RAM to swap.

Why this matters for Kafka:
- Kafka relies heavily on OS page cache for performance
- Swapping reduces available page cache
- Swap usage causes unpredictable latency spikes

Recommended behavior:
Keep data in RAM as long as possible.  
Setting swappiness close to 0 minimizes swap usage.


#### vm.dirty_background_ratio = 5
Defines when the kernel starts flushing dirty pages to disk asynchronously.

Dirty pages = data written to memory but not yet persisted to disk.

Why this matters:
- Kafka produces continuous disk writes
- Early background flushing prevents sudden I/O bursts
- Helps maintain stable write latency

Effect:
At ~5% dirty memory, Linux begins background disk writes gradually.


#### vm.dirty_ratio = 60–80
Defines the maximum percentage of system memory allowed to hold dirty pages before processes are forced to block and flush data.

Why this matters:
- Higher value allows larger sequential disk writes
- Reduces forced synchronous flush operations
- Improves throughput for append-heavy workloads

Trade-off:
Higher ratio = better throughput but longer recovery window after crash.

Kafka benefit:
Supports large, efficient sequential disk writes.


#### vm.max_map_count = 400000–600000
Controls how many memory-mapped files a process can use.

Why Kafka needs this:
- Each log segment file may be memory-mapped
- Many partitions → many segments → many mappings
- Low limits cause broker startup failure or instability

Production impact:
Higher value ensures large clusters can open all segment files reliably.


### JVM Tuning

Kafka relies more on OS memory than JVM heap.  
The goal is **stable GC behavior and predictable latency**, not large heap size.


#### -XX:+UseG1GC
Enables the G1 garbage collector.

Why G1GC is recommended:
- Designed for large heaps with predictable pause times
- Performs incremental collection
- Reduces long stop-the-world pauses
- Suitable for server workloads with continuous allocation

Kafka benefit:
More stable latency under sustained load.


#### -Xmx6g and -Xms6g
Sets JVM maximum and initial heap size to the same value.

Why equal heap sizes are recommended:
- Prevents heap resizing during runtime
- Avoids allocation pauses caused by heap expansion
- Produces more predictable GC behavior

Why Kafka uses moderate heap sizes:
- Kafka stores data on disk, not in heap
- OS page cache provides read performance
- Large heap reduces memory available for page cache

Typical production approach:
Use moderate heap (4–8 GB) and allocate remaining RAM to the operating system.


### Key takeaway

Kafka performance improves when:
- Linux page cache is maximized
- Disk writes are smooth and sequential
- JVM GC pauses are predictable
- Swap usage is minimized

Correct OS tuning often produces larger gains than JVM tuning.

## 8. Production Readiness Checklist — Detailed Explanation

Before putting Kafka into production, certain configuration and operational practices ensure **durability, availability, and predictable performance**.

### Replication factor ≥ 3
- Each partition should have at least 3 replicas.
- Ensures that the cluster can tolerate a broker failure without data loss.
- Provides leader election flexibility.
- Reduces risk of losing all copies during maintenance or crash.


### min.insync.replicas = 2
- Defines the minimum number of replicas that must acknowledge a write to consider it successful.
- Works together with producer setting `acks=all` for durability.
- Prevents data loss if a broker or network fails during write operations.

### Producers use `acks=all`
- Producers wait for all in-sync replicas to confirm the write.
- Guarantees strong consistency and durability.
- Slightly higher latency but critical for reliable production workloads.
- Must match `min.insync.replicas` to avoid rejected writes.


### Avoid swap
- Kafka performance relies heavily on **OS page cache**.
- Swap usage can reduce available cache and introduce unpredictable latency.
- Keep `vm.swappiness` low (e.g., 1) and ensure enough RAM for cache.


### Monitor GC, network, replica lag
- Track **garbage collection pauses** in the JVM for unexpected latency.
- Monitor **network utilization** to prevent bottlenecks in replication or consumer traffic.
- Watch **replica lag** to detect slow followers and potential ISR shrink.
- Use metrics and alerting to detect early signs of performance degradation.

### Key takeaway
A production-ready Kafka cluster combines:
- Proper replication and durability settings
- Memory and swap management
- Continuous monitoring of GC, network, and replication
- Safe operational practices to minimize downtime and data loss
